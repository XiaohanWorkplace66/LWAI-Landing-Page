§ 01 / The problem

Robot data is stranded, lossy, and un-versioned.

Fleets generate terabytes per day across eight sensor modalities. Today, most of it is written to local SSDs, copied by hand, and replayed in bespoke scripts that break on every schema change. Physical AI needs infrastructure, not glue.

0%

Of robotics eng time goes to data plumbing.

Every autonomy team rebuilds the same capture → sync → replay stack. That's engineering hours not spent on the model, the controller, or the robot itself.

0 in 6

Slices lost to desync or offline edge.

Multi-modal recordings drift by milliseconds, edge devices drop frames on reconnect, and the one frame you needed for the post-mortem is the one that's missing.

§ 02 / The platform

Four planes, one synchronized record.

DataCore is a single binary deployed in four coordinated planes. Each plane owns one stage of the lifecycle — capture, index, serve, replay — and speaks one time-aligned protocol. No glue scripts. No lost frames.

§ PLANE 01 · CAPTURE
dc capture --edge --offline-tolerant
Plane 01 · Capture

Edge-first, offline-tolerant recording.

One agent per robot captures every configured sensor simultaneously, tags each frame with a monotonic clock, and streams to the depot on reconnect — or keeps going if no network is ever available.

  • 8 modalities · ROS2 / ROS1 / raw UDP
  • Wall-clock + monotonic dual-timestamp
  • Zero-drop buffer to local NVMe, 48 h autonomous
  • H.265 camera, LZ4 telemetry, 3.1× average compression
# per-robot agent
dc capture start --robot=r-0182 --modalities=all
§ PLANE 02 · INDEX
dc index build --sync-tolerance=2ms
Plane 02 · Index

Timestamp-synced content index.

The index plane ingests raw streams and builds a sparse, queryable record per frame — hash, schema version, spatial bucket, modality, offset. Synchronization tolerance is explicit (default 2 ms).

  • Columnar store on S3 / GCS / MinIO
  • Content-addressed; de-duplicates across fleet
  • Schema evolution with deterministic downgrade
  • Spatial index for geofenced scenarios
# build index across a fleet
dc index build --fleet=sf-depot-01 --since=2026-03-01
§ PLANE 03 · SERVE
dc retrieve slice --t0/--t1 — p99 240ms
Plane 03 · Serve

Synchronized retrieval, under 240 ms.

Given any timestamp range and any modality set, the serve plane returns a time-aligned slice. No "approximately aligned" — the index guarantees every returned frame lies within the requested sync tolerance.

  • 240 ms p99 over 2.3 PB · 14 fleets
  • gRPC + Arrow Flight · zero-copy client
  • Partial slices for streaming model training
  • Budget-aware query planner
# 2-second pre-failure slice
dc retrieve slice --robot=r-0182 --t0=-2s --t1=0s
§ PLANE 04 · REPLAY
dc replay run --deterministic --tag=v23
Plane 04 · Replay

Deterministic re-execution.

Every slice is a first-class versioned dataset. Replay runs bit-exact against any model / env combination — the only way to ship a controller you can actually debug six months from now.

  • Content-hashed datasets · tag and branch
  • Replay into sim or against a recorded shadow-mode
  • Diff two replays across any modality
  • Export to TFRecord, WebDataset, or Parquet
# deterministic sim replay
dc replay run --id=run_014b3 --speed=0.5

Edge → Cloud → Tooling. One binary, three surfaces.

DataCore deploys as a single agent plus a single cloud service plus a single toolchain — no glue, no bespoke ingest, no per-robot bash scripts. The three surfaces speak one protocol.
